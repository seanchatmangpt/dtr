# Integrating DocTester CLI with Maven Projects

If you're developing Java applications with Maven, DocTester CLI helps you manage and publish the documentation generated by your Maven builds. This tutorial shows you how to integrate DocTester CLI into your Maven workflow.

## Why Integrate with Maven?

When you run `mvn clean verify`, DocTester (the Java library) generates HTML documentation in `target/docs/` or `target/site/doctester/`. DocTester CLI helps you:

- **Convert** those exports to Markdown for GitHub
- **Generate** coverage reports
- **Publish** to GitHub Pages or S3 automatically
- **Organize** documentation across multiple test suites

## Prerequisites

You'll need:
- A Maven project with DocTester (Java library)
- Java 25+ and Maven 4.0.0+ installed
- DocTester CLI installed (see [Getting Started](01-getting-started.md))

## Step 1: Generate Documentation with Maven

In your Maven project, run:

```bash
cd /path/to/your/maven/project
mvnd clean verify
```

This runs your tests and generates documentation. You'll see output in:

```
target/docs/
├── index.md
├── test-results/
│   ├── MyDocTest.md
│   └── AnotherTest.md
└── openapi.json
```

The documentation is in Markdown, but let's enhance it with DocTester CLI.

## Step 2: Use DocTester CLI to Manage Exports

Navigate to the directory containing your Maven documentation:

```bash
cd /path/to/your/maven/project/target/docs
```

### List Exports

See what documentation was generated:

```bash
dtr export list .
```

Output:
```
📁 Documentation Exports:
  ✓ index.md (3.2 KB)
  ✓ test-results/MyDocTest.md (12.5 KB)
  ✓ test-results/AnotherTest.md (8.3 KB)
  ✓ openapi.json (4.1 KB)

Total: 4 files, 28.1 KB
```

### Generate Reports

Create a summary report:

```bash
dtr report sum . -o summary.html
```

This generates an HTML summary showing:
- Number of test cases documented
- Documentation coverage
- Key API endpoints
- Code examples count

Generate a changelog (for multiple test runs):

```bash
dtr report log . -o changelog.md
```

## Step 3: Convert to Markdown (Recommended)

If your docs include HTML, convert them to Markdown for better GitHub integration:

```bash
# Convert any HTML files to Markdown
dtr fmt md . -o ../docs -r --force
```

The `-r` flag processes directories recursively. Now you have clean Markdown documentation in `docs/`:

```
docs/
├── index.md
├── test-results/
│   ├── MyDocTest.md
│   └── AnotherTest.md
└── openapi.json
```

## Step 4: Validate Exports

Before publishing, validate that your exports are intact:

```bash
dtr export check .
```

Output:
```
✓ Checking exports in: target/docs/
✓ All files present and valid
✓ Markdown syntax: OK
✓ Links: OK
✓ Code blocks: OK

Export integrity: PASSED
```

## Step 5: Archive Old Exports (Optional)

To keep things clean, archive older documentation:

```bash
# Archive exports older than 7 days
dtr export clean . --older-than 7d

# Or archive everything into a tar.gz
dtr export save . -o docs-backup.tar.gz
```

This is useful when running many tests and want to avoid clutter.

## Complete Maven Integration Workflow

Here's a complete workflow combining Maven and DocTester CLI:

```bash
#!/bin/bash
# Full build and documentation pipeline

echo "1. Running Maven build and tests..."
mvnd clean verify

echo "2. Listing generated documentation..."
dtr export list target/docs

echo "3. Converting to Markdown..."
dtr fmt md target/docs -o docs -r --force

echo "4. Generating coverage report..."
dtr report cov target/docs -o docs/coverage.html

echo "5. Validating exports..."
dtr export check target/docs

echo "6. Creating backup..."
dtr export save target/docs -o backups/docs-$(date +%Y%m%d).tar.gz

echo "✓ All done! Documentation ready in docs/"
```

Save this as `build-docs.sh`, make it executable, and run it:

```bash
chmod +x build-docs.sh
./build-docs.sh
```

## CI/CD Integration

Use this in your GitHub Actions workflow (`.github/workflows/docs.yml`):

```yaml
name: Build Documentation

on:
  push:
    branches: [main]

jobs:
  docs:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3

      - name: Set up Java
        uses: actions/setup-java@v3
        with:
          java-version: '25'
          distribution: 'openjdk'

      - name: Run Maven tests
        run: mvnd clean verify

      - name: Set up Python
        uses: actions/setup-python@v4
        with:
          python-version: '3.12'

      - name: Install DocTester CLI
        run: |
          cd doctester-cli
          pip install -e .

      - name: Convert and manage docs
        run: |
          dtr fmt md target/docs -o docs -r --force
          dtr export check target/docs

      - name: Publish to GitHub Pages
        if: success()
        run: |
          dtr push gh docs --repo ${{ github.repository }}
        env:
          GITHUB_TOKEN: ${{ secrets.GITHUB_TOKEN }}
```

This automatically:
1. Runs Maven tests
2. Generates DocTester documentation
3. Converts to Markdown
4. Publishes to GitHub Pages

## Common Maven + DocTester CLI Tasks

### Convert All Test Results to Markdown

```bash
dtr fmt md target/docs -o markdown_docs -r
```

### Publish Documentation to GitHub Pages

```bash
export GITHUB_TOKEN=your_token
dtr push gh target/docs --repo owner/repo
```

### Publish to AWS S3

```bash
export AWS_ACCESS_KEY_ID=your_key
export AWS_SECRET_ACCESS_KEY=your_secret
dtr push s3 target/docs --bucket my-docs --region us-west-2
```

### Archive and Clean Up

```bash
# Archive before cleaning
dtr export save target/docs -o backups/docs.tar.gz

# Remove old exports (older than 30 days)
dtr export clean target/docs --older-than 30d
```

## Organizing Multiple Maven Modules

If your project has multiple Maven modules, each generates docs:

```
multi-module-project/
├── module-a/target/docs/
├── module-b/target/docs/
└── module-c/target/docs/
```

Collect them into a single documentation site:

```bash
#!/bin/bash
mkdir -p consolidated-docs

# Copy and organize
dtr export list module-a/target/docs -o consolidated-docs/module-a/ -r
dtr export list module-b/target/docs -o consolidated-docs/module-b/ -r
dtr export list module-c/target/docs -o consolidated-docs/module-c/ -r

# Create index linking to all modules
cat > consolidated-docs/index.md << 'EOF'
# API Documentation

- [Module A](module-a/index.md)
- [Module B](module-b/index.md)
- [Module C](module-c/index.md)
EOF

# Publish
dtr push gh consolidated-docs --repo owner/repo
```

## Troubleshooting

**"target/docs not found"**
- Make sure you ran Maven first: `mvnd clean verify`
- Check the output directory in your `pom.xml` configuration

**"Export check fails"**
- Run with verbose: `dtr export check . -v`
- Check for broken links or missing files
- Regenerate docs: `mvnd clean verify`

**"GitHub Pages push fails"**
- Verify `GITHUB_TOKEN` is set: `echo $GITHUB_TOKEN`
- Use a token with `pages` and `repo` scopes
- Check repository settings enable GitHub Pages
