## The Documentation Drift Problem ŌĆö Root Cause Analysis

Documentation fails at scale. The diagnosis is almost always the same: the documentation and the system are separate artefacts with no enforcement boundary between them. A comment can say anything. A README can promise anything. Neither is checked by the compiler, the type system, or the test suite. The result is documentation drift ŌĆö the gradual divergence between what a system does and what its documentation claims it does.

The failure taxonomy below covers every major category of documentation drift observed across Fortune 500 engineering organisations. In every case, the root cause is the same: documentation is produced by humans reading code, not by machines executing code.

| Failure Mode | Root Cause | Consequence at Scale |
| --- | --- | --- |
| Stale inline comments | Refactor doesn't update comments | Engineers distrust all comments |
| Out-of-sync README | No CI check enforces README accuracy | Onboarding time doubles |
| Contradicting OpenAPI spec | Spec hand-authored, not generated | Client SDKs break silently |
| Missing error codes | Error handling added without doc update | Support tickets for known errors |
| Wrong performance claims | Benchmarks run once, doc never updated | SLA violations in production |
| Outdated auth flows | Security change not propagated to docs | Security incidents from bad clients |

> [!WARNING]
> Documentation drift is not a process problem. No amount of documentation review, style guides, or linting will solve it. It is a type system problem. The only lasting fix is to make documentation a typed output of test execution ŌĆö which is precisely what DTR provides.

## 2. sayCode() ŌĆö Executable Specification

Code blocks in documentation serve two purposes: they show readers what to type, and they show the test suite what the system must produce. `sayCode()` renders syntax-highlighted, language-tagged fenced blocks suitable for any Markdown renderer, static site generator, or documentation platform.

The example below shows the canonical implementation of a compensation handler for the saga pattern documented above. It uses Java 26's sealed classes and pattern matching to guarantee that every saga step has a compensation defined ŌĆö at compile time, not at runtime.

```java
// Saga step as a sealed hierarchy ŌĆö no step can exist without a compensation
sealed interface SagaStep permits
    SagaStep.ReserveInventory,
    SagaStep.ChargePayment,
    SagaStep.DispatchFulfillment,
    SagaStep.SendNotification,
    SagaStep.WriteAuditLog {

    // Every step MUST define its compensation. No default. Compiler-enforced.
    SagaStep compensate();

    record ReserveInventory(String sku, int qty) implements SagaStep {
        public SagaStep compensate() { return new ReleaseReservation(sku, qty); }
    }
    // ... all 5 steps defined, all 5 compensations defined
    // The compiler rejects any SagaStep that omits compensate()
}
```

> [!WARNING]
> Misuse pattern: pasting code into `sayCode()` that is not actually tested. This recreates documentation drift inside DTR itself. Every code block should be extracted from a real class or generated by a `sayCodeModel()` call.

## 3. sayWarning() and sayNote() ŌĆö Severity-Classified Assertions

Most documentation tools have at most one callout type. This is insufficient for enterprise systems where the distinction between 'you must know this' (WARNING) and 'this is useful context' (NOTE) determines whether a reader avoids a production incident or merely learns something interesting.

DTR maps directly to GitHub's native alert syntax, which is rendered with colour and iconography in GitHub, GitLab, and Gitea without any plugin or extension.

> [!WARNING]
> WARNING semantics: the reader MUST act on this information or risk a correctness failure, security vulnerability, or data loss. Use sparingly. If everything is a warning, nothing is.

> [!NOTE]
> NOTE semantics: the reader SHOULD read this for better understanding but can safely skip it without causing harm. Use for Erlang analogies, historical context, performance observations, and design rationale.

| Severity | When to Use | Erlang Analogy |
| --- | --- | --- |
| WARNING | Breaking change, security concern, data hazard | exit(Pid, kill) ŌĆö non-ignorable |
| NOTE | Context, rationale, cross-reference | logger:notice/2 ŌĆö informational |
| (Neither) | Normal prose that informs but does not alarm | io:format/2 ŌĆö routine output |

## Summary: Architectural Decision Guide for say* Methods

Choosing the wrong documentation method is not merely aesthetic ŌĆö it communicates incorrect intent. A list where order matters but `sayUnorderedList()` was used tells the reader 'sequence is irrelevant' when it is not. The table below encodes the decision criteria.

| If you need to document... | Use | Not |
| --- | --- | --- |
| A comparative claim | sayTable() | sayUnorderedList() |
| An ordered procedure | sayOrderedList() | sayUnorderedList() |
| An unordered set of requirements | sayUnorderedList() | sayOrderedList() |
| A live data structure | sayJson() | sayCode() with pasted JSON |
| Configuration / metadata | sayKeyValue() | sayCode() with properties file |
| Validated system properties | sayAssertions() | sayNote() with bullet points |
| A must-act warning | sayWarning() | sayNote() |
| Structural code facts | sayCodeModel() | sayCode() with hand-typed class |

> [!NOTE]
> The 'Not' column documents anti-patterns that produce plausible-looking documentation with incorrect semantics. The most common: using `sayCode()` with hand-pasted JSON instead of `sayJson()` with a live object. The pasted JSON will drift. The live object cannot.
